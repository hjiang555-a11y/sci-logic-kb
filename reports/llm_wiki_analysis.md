## 一、LLM Wiki 核心思想总结

### 1.1 核心差异：持久化编译 vs 每次检索
- **传统 RAG**：每次查询时从原始文档中检索相关片段，LLM 重新合成答案。无积累，重复劳动。
- **LLM Wiki**：LLM 增量构建并维护一个持久化的 wiki（结构化、互联的 Markdown 文件集合）。知识被**编译一次**，然后**保持更新**，而不是每次查询重新推导。

### 1.2 三层架构
1. **原始源 (Raw sources)**：不可变的源文档（论文、文章、图片等）。
2. **Wiki 层**：LLM 生成和维护的 Markdown 文件（实体页、概念页、摘要等）。
3. **模式层 (Schema)**：指导 LLM 如何构建和维护 wiki 的规范文档（如 CLAUDE.md）。

### 1.3 三大操作
- **摄取 (Ingest)**：处理新源 → 更新 wiki 页面、索引、日志。
- **查询 (Query)**：基于 wiki 回答问题，并将有价值的答案存回 wiki。
- **Lint**：定期健康检查，发现矛盾、过时声明、孤立页面、缺失链接等。

### 1.4 两个关键辅助文件
- **index.md**：内容目录，列出所有页面及其摘要、分类，便于 LLM 快速查找。
- **log.md**：时间线日志，记录摄取、查询、Lint 等活动，使用可解析的格式（如 `## [2026-04-02] ingest | 文章标题`）。

### 1.5 工具与技巧
- **搜索工具**：当 wiki 规模增大时，可使用本地搜索引擎（如 qmd）。
- **Obsidian 集成**：Web Clipper、本地图片下载、Graph View、Dataview、Marp 幻灯片等。
- **Git 版本控制**：Wiki 作为 Markdown 文件的 Git 仓库，天然具备版本历史和协作能力。

### 1.6 为什么有效
- **维护成本接近零**：LLM 不厌烦、不遗忘，可同时更新多个文件，保持 wiki 一致性。
- **人类角色**： curation（选源）、指导分析、提出好问题、思考意义。
- **LLM 角色**：所有簿记工作（总结、交叉引用、归档、保持一致性）。

## 二、sci-logic-kb 现状对照

### 2.1 当前架构
| 层次 | sci-logic-kb 实现 | 对应 LLM Wiki 概念 |
|------|-------------------|---------------------|
| **原始源** | Zotero 管理的 PDF 论文（Windows 存储） | Raw sources |
| **Wiki 层** | YAML 节点文件（机器可读）+ Obsidian Markdown 节点文件（人类可读） | Wiki（Markdown 文件） |
| **模式层** | `SCHEMA.md`（详细规范） + `CLAUDE.md`（行为规范） | Schema（CLAUDE.md） |
| **摄取流程** | 单篇论文处理流程（读取 PDF → 提取 YAML → 写入文件 → 更新 SCHEMA 表） | Ingest |
| **查询** | 尚未系统化，依赖人工查找或 LLM 临时扫描文件 | Query |
| **Lint** | 无定期健康检查 | Lint |
| **索引** | 无全局索引文件，仅 SCHEMA.md 中的“已处理论文”表 | index.md |
| **日志** | 无统一日志，进度记录分散在会话记忆、QUEUE.md 中 | log.md |

### 2.2 优势
- **结构化程度高**：YAML 节点、关系、条件变量等定义清晰，便于机器推理。
- **专题体系完善**：`TOPICS.md` 定义了专题层次，支持多专题并行。
- **已有工作流**：论文处理流程成熟，与 Zotero、Obsidian 集成良好。
- **GitHub Copilot 优先**：明确了 AI 协作分工。

### 2.3 待改进点（从 LLM Wiki 视角）
1. **缺乏全局索引**：LLM 无法快速了解知识库全貌，查询时需要遍历所有 YAML 文件。
2. **缺乏活动日志**：难以追踪“最近处理了哪些论文”、“Schema 何时更新”等。
3. **无定期 Lint**：随着节点增多，可能出现矛盾、孤立节点、过期声明等问题。
4. **查询未闭环**：用户提问得到的答案未被保存，知识未积累。
5. **模式演化未文档化**：SCHEMA.md 虽为真源，但变更记录未系统化。
6. **搜索工具缺失**：当节点数超过数百时，线性查找效率低。

## 三、具体改进建议

### 3.1 立即可实施的改进（低风险、高价值）

#### 3.1.1 创建全局索引文件 `INDEX.md`
- **位置**：仓库根目录 `INDEX.md`
- **内容**：每个节点一行，包含：
  - 节点 ID（链接到 YAML 文件）
  - 节点类型（ent/pri/meth/met）
  - 专题（ultrastable-laser, optical-frequency-combs, ...）
  - 简要描述（从 YAML 的 `description` 提取）
  - 所在文件
- **生成方式**：编写 Python 脚本扫描 `topics/*/papers/*.yaml`，自动生成并更新。
- **用途**：LLM 查询时先读索引定位相关节点，大幅减少文件读取。

#### 3.1.2 创建活动日志 `LOG.md`
- **位置**：仓库根目录 `LOG.md`
- **格式**：每个条目以 `## [YYYY-MM-DD] 操作 | 详情` 开头，例如：
  ```
  ## [2026-04-20] ingest | matei2017.yaml (Ultra‑stable laser sub‑10 mHz linewidth)
  ## [2026-04-19] schema_update | v4.1 光学频率梳专题重构完成
  ## [2026-04-19] lint | 检查超稳激光专题，发现 2 个孤立节点
  ```
- **维护**：在论文处理流程的最后一步、Schema 更新后、Lint 运行后自动追加。
- **用途**：提供时间线，帮助 LLM 了解近期活动，便于人类追踪进度。

#### 3.1.3 设计 Lint 检查脚本
- **脚本位置**：`scripts/lint_kb.py`
- **检查项**：
  - 孤立节点（无任何关系）
  - 矛盾声明（同一指标在不同论文中数值冲突）
  - 过期声明（某原理已被新论文推翻或改进）
  - 缺失必填字段（如 `source.claim`）
  - 节点 ID 冲突（重复）
  - 跨专题引用是否正确
- **运行频率**：每周一次，或每次批量处理论文后。
- **输出**：报告文件 + 自动修复建议（部分可自动修复）。

### 3.2 中期改进（需要一些开发）

#### 3.2.1 增强查询流程
- **查询入口**：用户提问 → LLM 读取 `INDEX.md` 定位相关节点 → 读取对应 YAML 文件 → 合成答案。
- **答案存档**：若答案具有长期价值（如比较表、分析总结），将其保存为新的 YAML 节点或关系，并更新索引和日志。
- **实现方式**：在 `CLAUDE.md` 中增加“查询模式”章节，定义标准流程。

#### 3.2.2 模式演化日志
- 在 `SCHEMA.md` 开头添加“版本变更历史”节，记录每个版本的变更原因、影响范围。
- 每次更新 Schema 时，同时更新该日志。

#### 3.2.3 集成本地搜索工具
- 评估 [qmd](https://github.com/tobi/qmd) 是否适合我们的 YAML/Markdown 混合存储。
- 若适合，配置 qmd 作为 CLI 工具，供 LLM 调用。
- 若不合适，编写简易关键词搜索脚本（基于 `INDEX.md` 和文件内容）。

### 3.3 长期优化（可选）

#### 3.3.1 Obsidian 深度集成
- 利用 Dataview 插件，基于 YAML frontmatter 生成动态视图（如“所有指标列表”、“某个专题的所有原理”）。
- 配置 Marp 插件，直接从 wiki 生成演讲幻灯片。
- 优化 Graph View，确保节点连接清晰可见。

#### 3.3.2 自动化摄取流水线
- 监听 Zotero 数据库变化，自动触发论文处理流程。
- 结合 GitHub Actions，实现 CI/CD 式知识库更新。

#### 3.3.3 跨专题推理引擎
- 基于现有关系（BOUNDED‑BY, ENABLED‑BY, COMPETES‑WITH）实现简单推理，回答“为什么卡在这？”“如何突破？”等复杂问题。

## 五、风险与注意事项

1. **向后兼容**：所有改进不得破坏现有 YAML 文件的解析。
2. **性能**：索引生成和 Lint 检查在大规模（数千节点）时仍需高效。
3. **人工审核**：Lint 发现的“矛盾”可能只是不同实验条件所致，需人工判断。
4. **Git 冲突**：`INDEX.md` 和 `LOG.md` 可能因多人协作产生冲突，需设计合并策略。

## 六、结论

Karpathy 的 LLM Wiki 模式与 sci-logic-kb 目标高度契合，都强调**持久化、结构化、LLM 维护**的知识库。我们的项目已在结构化、专题化、工作流方面打下坚实基础，接下来可重点补全**索引、日志、Lint**三大辅助设施，并逐步实现**查询闭环**和**搜索增强**。

**建议立即启动阶段一**，先建立索引和日志，让 LLM 能更快、更准地了解知识库全貌，为后续智能查询打下基础。
